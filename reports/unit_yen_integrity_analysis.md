# unit_yen CSV / インポート / UI 不整合の整理（調査メモ）

対象: `fixed_csv_3/unit_yen/` の `10.csv`, `import_firstTime_105.csv`, `16_20251224.csv`, `30.csv` および `import_full_update_fast.py` の挙動。

---

## 1. `import_firstTime_105.csv`（本社と代表の住所・郵便、株主、J列、設立）

### 根本原因（コード）

`import_full_update_fast.py` の `resolve_current_map` は次のようにインデックスを組み立てています。

```python
norm_to_idx = {normalize_header(h): i for i, h in enumerate(headers)}
```

**同じ正規化ヘッダ名（例: `郵便番号`, `住所`）が複数あると、後ろの列インデックスだけが残ります。**

このファイルのヘッダは概ね次の並びです。

| 列(0始まり) | ヘッダ名 | 意味（仕様上） |
|------------|----------|------------------|
| 2 | 郵便番号 | **本社** |
| 3 | 住所 | **本社** |
| 6 | 郵便番号 | **代表者（G列相当）** |
| 7 | 住所 | **代表者（H列相当）** |

その結果、**本社の郵便・住所がマッピングから消え**、DB の `postal_code` / `address` には **6–7列目（代表者側）だけ**が入ります。  
一方 `代表者郵便番号` / `代表者住所` というヘッダ名は無いため、`representative_postal_code` / `representative_home_address` は **空のまま**になりやすく、画面上は「本社と代表が逆」「代表欄が空」などに見えます。

`株式保有率` はヘッダが一意なので **11列目は `shareholders` に正しく載る想定**ですが、列ずれしている行では別列の値が入り「株主に入っていない」ように見えることがあります。

`設立` と `創業` はどちらも `founding_year` にマップされる定義のため、**どちらか一方の列しか効かない**状態になり得ます（重複ヘッダと合わせて「設立が壊れている」ように見える要因）。

### 対応（実装済み）

ファイル名に `import_firsttime`（大文字小文字無視）を含む CSV は **`IMPORT_FIRSTTIME_INDEX_MAP` で列位置固定**するよう変更しました。

- 2–3: 本社 `postal_code` / `address`
- 6–7: 代表 `representative_postal_code` / `representative_home_address`
- 10: `設立` → `founding_year`
- 11: `株式保有率` → `shareholders`
- 9列目（ヘッダ空・Excel の J 列相当）: 事業構成テキストを **`overview` の末尾に `【事業構成】…` として追記**
- 8列目 `創業`: **`overview` に `【創業】…` として追記**（`設立` と混同しないため）

**再インポート**が必要です。

### 代表者誕生日

`import_firstTime_105.csv` に **代表者誕生日列はありません**。欠損はソースデータの制約です。

## 2. `10.csv`（標準41列）

- ヘッダは `郵便番号`（本社）と `代表者郵便番号` / `代表者住所` が **別名**のため、上記の「重複ヘッダ」問題は起きにくいです。
- **6行目（データ上5行目）**の例: `株式会社ティー・エー・エス` は CSV 上も `代表者誕生日` が空（`,,`）です。インポート結果が空なのは仕様通りの可能性が高いです。
- 「NDA が締結済なのに UI で有効でない」は CSV だけでは断定できません。DB の `nda_flag` と UI の表示条件を突合してください（`scripts/audit_csv_vs_db_integrity.py` 等）。

## 3. `16_20251224.csv` と売上 224628（淵野辺デベロップメント特定目的会社）

- 該当行はファイル **約2339行目**（法人番号 `6010005032676`）。「224628行目」はファイル行番号ではなく **売上欄の数値**と解釈するのが自然です。
- `直近決算年月` は CSV 上 `2023年1月1日` と入っています。UI に出ない場合は **フロントの項目名・フォーマット・NULL 扱い**、または **別レコード更新で上書き**を疑ってください。
- `import_full_update_fast.py` の `parse_revenue_profit` は、単位語（万・百万・億）が無い **純粋な数値に対して既定で 1000 倍**します。  
  `224628` → **224,628,000**（円）として格納される想定です。  
  「1000 倍されていない」と感じる場合は、**UI の表示単位（千円・百万円表示など）**か、**別経路で取り込んだ値**の可能性があります。DB の `latest_revenue` を直接確認するのが確実です。
